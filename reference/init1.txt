- AI는 만든다 — 문제를 이해했으니 뭔가 그럴듯한 것이 나온다
- 같은 문제를 이미 누가 더 잘 풀고 있는지 확인하지 않았다
- 다른 접근 방식이 있는지 탐색하지 않았다
- 이 솔루션이 기존 대안과 뭐가 다른지 정의하지 않았다
- 여러 가능성 중에서 왜 이걸 선택했는지 근거가 없다
- 이 솔루션이 최선인지 아무도 모른다

- 처음 떠오른 아이디어에 집착한다
- "이 기능이면 되겠지" — 그 기능이 정말 핵심 문제를 푸는 건지 아니면 내가 만들고 싶은 건지
- 더 간단하고 더 직접적인 방법이 있을 수 있다
- 솔루션을 구조적으로 탐색하고 선택하는 방법이 있다
  - 여러 가능성을 열어놓고 (발산)
  - 기준을 세워서 최선을 고른다 (수렴)

❓ HMW (How Might We?) 메모법
- 라이트닝 토크를 들으며 포스트잇에 아이디어를 기록해요.
- How: 기회가 있다고 가정합니다
- Might: 다양한 가능성을 열어둡니다
- We: 함께 해결한다는 뜻입니다

✍️ HMW 작성 요령
- 두꺼운 검정 펜으로 씁니다
- 간결하게 씁니다
- 한 포스트잇당 한 가지 아이디어
- 너무 넓지도, 너무 좁지도 않게
- 안 적으면 투표할 수 없습니다!

- 프로젝트 비전 / 비즈니스 목표
- 사용자 관점 / 사용자 조사 결과
- 현재 제품 분석 / 디자인 변화 과정
- 경쟁사 분석
- 기술 고려사항 및 기회
- 모든 주제를 다 다루지 않아도 되며, 문제 공간을 풍부하게 이해하는 데 도움이 되는 범위 내에서 선택하세요.

[비전 / 비즈니스 목표]

- 성공을 어떻게 정의하는가?
- 가장 큰 위험 요소는 무엇인가?

- 이상적인 사용자 경험 흐름은 어떤 모습인가?

[경쟁사 / 유사 서비스]
- 유사하거나 참조할만한 제품은 무엇인가?
- 경쟁사가 잘하고 있는 것, 못하고 있는 것은?
- 타 산업에서 배울 수 있는 점은 무엇인가?

[기술적 고려 사항]
- 웹, 모바일, 앱 중 어디에 해당하는가?
- 데이터는 어디서 오고 어떻게 쓰이는가?
- 주요 기술적 제약 사항은?

🗂 어피니티 매핑

- 초기 분류가 명확하지 않아도 OK — 겹치는 내용을 기준으로 시작
- 나중에 더 나은 분류로 수정 가능
- 목표: 유용한 카테고리 맵 만들기

🗳 투표

- 가장 중요하다고 생각하는 메모에 점을 붙인다 — 몰아도 되고 나눠도 된다
- 투표 수가 많은 메모가 우선 검토 대상
- 방향을 정하는 것이 아니라, 영향력 있는 아이디어를 걸러내는 과정

🗺 사용자 여정 그리기

- 문제점(페인 포인트)도 함께 표시한다

🎯 목표 및 성공 기준 설정
- 어떤 사용자에 초점을 맞출 것인가?
- 어떤 순간 또는 어떤 문제점을 해결하고 싶은가?
- 성공은 어떤 모습인가? 어떻게 측정할 수 있는가?

🔍 유사 문제 영역의 사례 찾기

- 기존 산업 외부에서 영감을 얻는다

🔄 Crazy 8's
- 종이를 8칸으로 접는다
- 각 칸에 하나의 아이디어를 1분 안에 스케치한다
- 스토리보드가 아니라 아이디어의 스냅샷
- 수량이 우선 — 완성도는 나중. 8가지 다른 접근에 도전한다

🎯 솔루션 스케치
- 최고의 아이디어 하나를 선택한다
- 여러 상태(단계)를 보여주는 스케치를 만든다
- 설명을 위한 단어도 포함한다 — 보지 않아도 이해되게
- 눈에 띄는 제목도 붙인다

🔍 가설 검증
- 내 아이디어에 숨겨진 가정은 무엇인가
- "사용자가 바로 이해할 것이다" — 정말?
- "이 기능이 있으면 매일 쓸 것이다" — 정말?
- 검증하고 싶은 질문 상위 3개를 고른다

📋 스토리보드
- 4~5개의 핵심 순간으로 압축한다
- 소프트웨어 밖에서 시작한다 — 사용자가 서비스를 만나기 전부터
- 감정을 중심으로 — 각 순간에서 사용자가 어떤 상태인가

🛠 프로토타입
- 완제품이 아니라 가설을 검증하기 위한 실험용 결과물
- 사용자에게 현실적으로 느껴질 정도면 충분하다
- 빠르게 만들고 빠르게 실패해보기 위한 도구

🧪 사용자 테스트
- 프로토타입을 실제 사용자에게 사용하게 하고 반응을 관찰한다
- 테스트 대상자는 실제 사용 가능성이 높은 사람이어야 한다
- "좋다/싫다"보다 왜 그런 반응을 보이는지를 파악한다
- 방어적으로 설명하지 않고, 관찰하거나 질문만 던진다

📝 학습
- 무슨 일이 있었는지 요약한다
- 가설이 맞았는가, 틀렸는가
- 다음에 무엇을 바꿔야 하는가
- 필요하면 다시 아이디어 단계로 돌아간다

- 디자인 사고를 활용한 문제 해결의 최적화
- 발산과 수렴을 적절하게 활용해 고객 문제를 해결하는 프로세스
- 검증된 고객 문제를 해결하는 최적의 솔루션을 찾기 위한 도구

실무에 적용할 때
- 처음에는 시행착오가 있을 수 있다
- 회차를 거듭할수록 좋아진다
- 제품의 지속적인 성장 속도 = 스프린트의 속도

- 만들 수 있게 되면 만들고 싶은 것이 넘쳐난다
- "이 기능도 있으면 좋겠다" → AI에게 시킨다 → 만들어진다
- "이것도 추가해볼까" → 시킨다 → 또 만들어진다
- 일주일 뒤 → 기능은 10개가 늘었는데, 제품이 나아졌는지 모르겠다

2026년 바이브코딩에서 반복되는 패턴
- AI를 쓰는 1인 창업자는 월 8~12개 기능을 출시
- 하지만 성공하는 상위 10%의 공통점은 "더 많이 만드는 것"이 아님
- 시간의 40~50%를 "무엇을 만들지 판단하는 데" 쓰는 것

"만드는 속도가 빨라질수록, 방향이 맞는지 판단하는 일이 더 중요해진다"

기능은 늘었는데 제품이 안 나아지는 현상
- 기능 10개를 추가했는데 사용자가 쓰는 건 여전히 2개
- 화면이 복잡해져서 처음 쓰는 사람이 핵심 기능을 찾지 못한다
- 여기저기 기능이 늘어났는데 전체가 하나의 서비스처럼 안 느껴진다

기술 부채와 유사한 기능 부채
- 에이전트는 시키지 않은 것은 신경 쓰지 않는다
- 제품의 방향을 아무도 안 보면 기능 부채가 쌓인다
- 쌓인 기능은 치우는 것이 추가하는 것보다 훨씬 어렵다

"기능을 추가하는 것은 AI가 해준다. 이 기능이 있어야 하는지 판단하는 것은 내가 해야 한다"

소프트웨어의 본질적 장점
- 자동화: 한 번 만들면 같은 작업을 반복해서 실행할 수 있다
- 확장성: 나 혼자 쓸 수도 있고, 같은 문제를 겪는 다른 사람도 쓸 수 있다

내가 해결한 문제는 나만의 문제가 아닐 가능성이 높다
- 비슷한 상황에 있는 사람에게 이 제품을 주면 그 사람의 문제도 해결된다
- 그런 사람이 많을수록 이 제품을 만든 가치가 커진다

그래서 "이 제품이 해결하는 문제"를 더 넓게 바라볼 필요가 있다
- 지금 해결하는 문제의 앞뒤 단계에도 불편함이 있지 않은가?
- 다른 상황에서 비슷한 문제를 겪는 사람은 없는가?

내가 불편한 것 ≠ 사용자가 불편한 것
- 내가 구상한 기능이 최선이라는 보장이 없다
- 같은 문제를 이미 다른 제품이 더 나은 방식으로 풀었을 수 있다
- 리서치만으로는 모든 것이 보이지 않는다

기술적으로 만들 수 없는 것일 수 있다
- 만들다가 "구조부터 뜯어야 합니다"를 만나면 일정이 통째로 밀린다
- 구조 변경은 엄청나게 큰 일이다

사용자에게 물어봐도 정답이 안 나올 수 있다
- "더 빠른 말이라고 답했을 것이다" — 헨리 포드
- 사용자는 불편한 건 말할 수 있지만, 해결책은 모른다

- 이 문제를 풀면 어떤 모습이어야 하는가? → 비전 — 방향 점검
- 다른 제품이 더 나은 방식으로 풀어놨을 수 있다 → 경쟁 — 다른 접근 확인
- 기술적으로 어떻게 만들 수 있는가 → 기술 — 가능/불가능 파악
- 사용자가 진짜 원하는 것은 내 생각과 다를 수 있다 → 고객 — 실제 목소리에서 문제 발견

이 4가지를 AI에게 분석시키고, 결과를 내 판단의 재료로 쓴다
- AI는 각 시각에서 정보를 모아주고 분석해준다
- 그 분석 결과가 맞는지 검증하는 것은 내가 한다
- 리서치 문서와 고객 프로필 = 더 나은 판단

일반적인 기능 제안의 방식: "알림 기능", "통계 페이지", "다크모드"

이걸 그대로 AI에게 시키면 어떻게 되는가
- "알림 기능 만들어줘" → 푸시 알림인지, 앱 내 뱃지인지, 이메일인지 AI가 임의로 고른다
- 왜 필요한지 모른 채 만들어진다 → 내가 원했던 것과 다른 결과

같은 의도를 "문제"로 적으면 달라진다
- ❌ "알림 기능 추가"
- ✅ "어떻게 하면 사용자가 중요한 변경사항을 놓치지 않게 할 수 있을까?"

HMW(How Might We) — "어떻게 하면 ~할 수 있을까?"
- AI에게 "이 문제를 해결하는 방법을 3가지 제안해줘"가 가능
- 해결책의 선택지가 열린다 — 푸시 알림 말고도 배너, 이메일 요약, 변경 로그 등

너무 넓으면 → AI가 방향을 못 잡는다
- ❌ "어떻게 하면 더 좋은 서비스를 만들 수 있을까?"
- ✅ "어떻게 하면 첫 방문자가 30초 안에 핵심 기능을 시작할 수 있을까?"

너무 좁으면 → 해결책이 하나로 고정된다
- ❌ "어떻게 하면 홈 화면 상단에 빨간 뱃지를 넣을 수 있을까?" ← 이미 해결책
- ✅ "어떻게 하면 사용자가 새로운 내용이 생겼음을 자연스럽게 알 수 있을까?"

해결책이 아니라 문제에 집중한다
- ❌ "어떻게 하면 카카오 알림톡을 연동할 수 있을까?"
- ✅ "어떻게 하면 앱을 열지 않아도 중요한 소식을 받을 수 있을까?"

이 문제가 해결되면 어떤 상태가 되는가?
- 내가 근본적으로 "해결하려는 문제"는 무엇이었는가?
- 그 문제가 완전히 해결된 상태를 상상하면 어떤 모습인가?

비전이 명확하면 "무엇을 안 만들지"가 보인다
- 이 서비스의 핵심 가치를 한 문장으로 말할 수 있는가?
- 그 한 문장에 비춰보면, 추가하고 싶었던 기능 중 곁가지가 보인다
- 곁가지를 치우는 것도 제품을 발전시키는 것이다

비전과 현실의 간극이 곧 HMW의 재료
- "처음에 이렇게 되길 바랐는데, 지금은 여기까지밖에 안 된다"
- 그 간극 하나하나가 HMW 질문이 된다

- 이 문제가 완전히 해결된 상태를 상상하면? "완성된 모습"은 어땠는가?
- "이것만 되면 완벽한데"라고 느끼는 순간이 있는가?
- 이 제품이 해결하는 문제의 "근본 원인"까지 건드리고 있는가, 아니면 증상만 다루고 있는가?
- 6개월 뒤 이 제품이 어떤 상태면 "잘 발전했다"고 느낄 것인가?

"이 서비스의 목적은 [서비스 개요의 핵심 문제]를 해결하는 것이야. 나는 이 제품이 [비전 설명] 모습이기를 기대해. 상상하는 완성된 모습과 현재 이해 사이의 간극을 분석하고, 이 간극을 줄이기 위해 제일 먼저 해결해야 할 문제들을 'How Might We ~?' 형식으로 10개 제안해줘."

확인할 것
- AI가 제안한 HMW 중 내 문제와 직결되는 것과 곁가지를 구분한다
- 핵심 문제와 직결되는 간극을 우선으로 표시한다
- 서비스 개요의 핵심 문제와 직결되는 간극을 우선으로 표시한다
- 미처 생각하지 못한 관점이 포함되어 있는가?

같은 문제를 해결하려는 제품은 이미 시장에 존재한다
- 완전히 새로운 문제란 거의 없다
- 경쟁 제품들은 이미 여러 시행착오를 거쳤다

왜 그 제품을 쓰지 않고 직접 만들기로 했는가?
- 그 제품이 내 문제를 충분히 해결해주지 못했기 때문이다
- 그 "부족한 지점"이 바로 내 제품의 차별점이 될 수 있다

경쟁 분석에서 봐야 하는 것
- 기능 목록이 아니라 → 그 기능이 어떤 문제에 대한 답인지
- "이 제품은 알림 기능이 있다"가 아니라 → "이 제품은 사용자가 변경사항을 놓치는 문제를 푸시 알림으로 풀었다"
- 잘 만든 제품들의 공통 방식이 곧 사용자의 멘탈 모델이다

"경쟁 분석은 카피하기 위한 것이 아니라, 같은 문제를 더 잘 풀 수 있는 기회를 찾는 것이다"

- 이 제품과 같은 문제를 해결하려는 서비스 3개를 꼽는다면 무엇인가?
- 그 서비스들은 이 문제를 어떤 방식으로 풀었는가? (기능이 아니라 접근 방식)
- 그 접근 방식 중 "이건 잘했다"고 느끼는 부분은? 왜?
- 반대로 "이 부분이 불편해서 내가 직접 만들었다"고 느끼는 부분은?
- 경쟁 제품 사용자들이 자주 불만을 표현하는 지점은 어디인가?

"내가 풀려는 문제는 [문제 설명]이고, 비슷한 문제를 해결하는 경쟁 서비스로는 [경쟁 서비스 이름들]이 있어. 이 경쟁 서비스들의 주요 기능과 접근 방식을 분석하고, 다음을 정리해줘:
- 경쟁 서비스들이 공통적으로 잘하고 있는 것
- 경쟁 서비스들에서 사용자들이 자주 불만을 느끼는 것
- 내 서비스가 차별화할 수 있는 기회 영역
그리고 이 분석을 바탕으로 'How Might We ~?' 형식의 개선 아이디어를 10개 제안해줘."

확인할 것:
- 단순히 경쟁사 기능을 따라하는 방향이 아닌가?
- 내 사용자에게도 해당되는 문제인가? — 경쟁사 사용자와 내 사용자는 다를 수 있다
- 멘탈 모델(사용자가 익숙한 방식)과 충돌하지 않는가?
- 경쟁사가 못하고 있는 것 중 내 서비스의 정체성과 맞는 것이 있는가?

"나와 비슷한 문제를 겪는 사람들"은 이미 존재한다.
- 내가 해결하려는 문제로 힘들어하는 사람들이 분명히 어딘가에 있다
- 그들은 커뮤니티, 포럼, SNS, 리뷰 등에서 자신의 어려움을 이야기하고 있다

검색을 통해 실제 사용자들의 목소리를 수집하자
- "이런 문제 때문에 힘들다"
- "이런 기능이 있으면 좋겠다"
- "이 서비스 써봤는데 이 부분이 불편하다"

이 목소리들이 제품 개선의 가장 확실한 근거가 된다
- 내 생각이 아니라 실제 사람들이 겪는 문제다
- 여러 사람이 같은 말을 하면 우선순위가 올라간다

- 이 문제를 겪는 사람들이 주로 모이는 커뮤니티나 포럼은 어디인가?
- 그곳에서 사람들이 가장 자주 호소하는 어려움은 무엇인가?
- 경쟁 서비스의 리뷰에서 반복적으로 나오는 불만은 무엇인가?
- 이 문제를 겪는 사람들이 현재 어떤 방식으로 해결하고 있는가? (대안, 우회 방법)
- 관련 키워드로 검색했을 때 사람들이 묻는 질문들의 패턴은?

"리서치 문서와 고객 프로필을 바탕으로 타겟 사용자가 누구인지 파악해고 이들이 온라인에서 실제로 어떤 어려움을 호소하는지 찾아줘:
- 관련 커뮤니티, 포럼, SNS에서 자주 나오는 불만이나 질문
- 비슷한 서비스들의 리뷰에서 반복되는 피드백
- 이 문제를 겪는 사람들이 현재 사용하는 대안이나 우회 방법
- 사람들이 "이런 게 있으면 좋겠다"고 말하는 것들
수집한 내용을 바탕으로 'How Might We ~?' 형식의 개선 아이디어를 10개 제안해줘."

확인할 것:
- AI가 찾은 불만이 앞서 정의한 페르소나의 맥락과 맞는가?
- 그 불만이 JTBD(사용자가 끝내려는 일)와 연결되는가?
- AI가 추측한 문제인가, 실제 사람의 목소리에서 나온 것인가? — 출처를 확인한다
- 여러 곳에서 반복적으로 나오는 패턴이 있는가? — 반복되는 것이 우선순위가 높다

아이디어가 아무리 좋아도 만들 수 없으면 소용없다
- "이 기능 만들어줘" → "이건 구조를 먼저 바꿔야 합니다" → 일정이 통째로 밀린다
- 구조 변경은 엄청나게 큰 일이다

AI를 기술 전문가로 활용한다
- 이 문제를 해결하는 데 필요한 기술을 분석하고, 가능한 것과 어려운 것을 나눈다
- 바로 만들 수 있는 것 / 구조 변경이 필요한 것 / 현실적으로 어려운 것
- 이 구분이 우선순위를 정할 때 실현 가능성 필터 역할을 한다

기술 전문가의 눈으로 검토하면 보이는 것
- 같은 문제를 해결하는 데 더 적합한 기술이나 패턴이 있는가?
- 현재 방식의 한계는 무엇이고, 그 한계를 넘으려면 어떤 변경이 필요한가?
- 지금 쓰고 있는 기술 스택에서 아직 활용하지 않는 기능이 있는가?

- 이 문제를 해결하는 데 일반적으로 사용되는 기술적 접근법에는 어떤 것들이 있는가?
- 현재 내가 선택한 방식의 장단점은 무엇인가? 다른 방식과 비교하면?
- 지금 구조에서 확장하기 어려운 부분이 있다면, 기술적으로 어떤 대안이 있는가?
- 비슷한 문제를 해결한 다른 서비스들은 기술적으로 어떻게 접근했는가?

"리서치 문서와 서비스 구상을 바탕으로 [이런 문제]를 해결하기 위해 제품을 만들기 위해 기술 전문가 관점에서 검토해줘:
- 이 문제를 해결하는 데 일반적으로 사용되는 기술적 접근법들
- 내가 선택한 방식의 장점과 한계
- 같은 문제를 더 효과적으로 해결할 수 있는 대안적 접근법
- 최근 이 영역에서 주목받는 기술이나 패턴
그리고 기술적 관점에서 'How Might We ~?' 형식의 개선 아이디어를 10개 제안해줘."

확인할 것:
- 기술 변경 자체가 목적이 아니라, 문제 해결의 질을 높이는 방향인가?
- AI가 제안한 기술적 접근이 현실적인지 검증한다
- "바로 가능 / 구조 변경 필요 / 현실적 어려움"이 명확히 구분되는가?

AI가 잘하는 것
- 정보를 빠르게 모아준다 — 4가지 시각에서 수십 개의 분석 결과를 만들어준다
- 패턴을 찾아준다 — 흩어진 정보에서 공통점을 뽑아준다
- HMW 형식으로 정리해준다 — 문제를 구조화하는 데 능숙하다

AI가 못하는 것
- 내 사용자를 아는 것 — AI는 일반적 사용자를 상상할 뿐이다
- 내 맥락을 아는 것 — 리서치 과정에서 내가 직접 파악한 것을 AI는 모른다
- 무엇이 중요한지 판단하는 것 — AI는 모든 HMW를 동등하게 취급한다

전통적 개발에서는 만드는 데 시간이 걸렸다
- 기능 하나를 만드는 데 몇 주가 걸렸으니, 자연스럽게 "이걸 정말 만들어야 하는가?"를 고민했다
- 만드는 비용이 높으니까, 잘못된 방향으로 갔을 때 손해가 크다는 걸 알았다

바이브코딩에서는 만드는 비용이 거의 0이다
- AI에게 시키면 몇 시간 만에 만들어진다
- 그래서 "일단 만들어보자"가 됨 → 판단 없이 기능이 쌓인다
- JP Morgan 보고서의 표현: "3번 시도할 수 있던 것을 33번 시도할 수 있다"
- 하지만 33번 시도할 수 있다는 건 → 33번의 판단이 필요하다는 뜻이기도 하다

"더 많은 시도 기회"를 낭비하지 않으려면
- 시도할 때마다 "왜 이걸 시도하는가"의 근거가 있어야 한다
- 33번의 시도를 감이 아니라 판단으로 쓰는 사람이 이긴다

좋은 우선순위 선정을 위해 AI와 함께 이 질문에 대해 답을 정리해본다
- 이 문제에 대한 이해가 어느 수준에 있는가?
- 아직 검증되지 않은 가정이 있다면 무엇인가?
- 이 문제에서 가장 심각한 부분과 상대적으로 덜한 부분은?
- 지금 가장 시급하게 해결해야 할 문제는 무엇이라고 생각하는가?

기능이 많아지는 것이 아니라, 쓰이는 기능을 만드는 것이 관건이다
- SaaS 기업들이 사용되지 않는 기능에 매년 낭비하는 비용이 기업당 평균 1,800만 달러 (Zylo 2026)

솔루션을 만들기 전에 문제를 깊이 이해해야 한다 - 사용자 여정 지도(User Journey Map)란?
- 사용자가 특정 목표를 달성하기까지 겪는 전체 경험을 시각화한 것
- 단계별로 사용자의 행동, 생각, 감정을 정리한다
- 어느 지점에서 불편함을 느끼는지 파악할 수 있다

왜 필요한가?
- HMW 질문만으로는 문제의 맥락이 보이지 않는다
- "알림을 놓친다"는 문제가 언제, 왜 발생하는지 알아야 한다
- 전체 흐름에서 문제 지점을 찾아야 효과적인 솔루션이 나온다

"이 문제를 겪는 사용자가 [핵심 목표]를 달성하는 전체 여정을 분석해줘.
다음 형식으로 사용자 여정 지도를 만들어줘:
- 단계: 사용자가 거치는 주요 단계 (5-7단계)
- 행동: 각 단계에서 사용자가 하는 구체적 행동
- 생각: 그 순간 사용자의 머릿속 생각
- 감정: 긍정적(+) / 중립(0) / 부정적(-) 중 하나
- 불편 지점: 사용자가 불편함을 느끼는 순간
- 기회 영역: 개선할 수 있는 지점
특히 아래 HMW 질문들이 여정의 어느 지점과 연결되는지 표시해줘."

확인할 것:
- 여정이 실제 사용 패턴과 맞는가?
- 불편 지점이 HMW 질문들과 연결되는가?
- 놓치고 있던 문제 지점이 새로 보이는가?

어피니티 매핑(Affinity Mapping)이란?
- 비슷한 아이디어들을 그룹으로 묶는 기법
- 흩어진 아이디어들 사이에서 패턴을 발견한다
- "이것들은 결국 같은 문제를 다르게 표현한 것이구나"

에이전트 투표
- 디자인 스프린트에서는 팀원들이 스티커로 투표한다
- 우리는 AI에게 다양한 전문가 역할을 맡겨 투표하게 한다
- 제품 관리자, 개발자, 디자이너, 사용자 관점에서 각각 평가

왜 이렇게 하는가?
- 내 관점에만 갇히지 않기 위해
- 다양한 이해관계자의 시각을 시뮬레이션
- 편향 없이 우선순위를 정하기 위해

"다음 HMW 아이디어들을 분석해줘:
[HMW 리스트 붙여넣기]
먼저 비슷한 주제끼리 그룹으로 묶어줘.
그 다음, 4가지 관점에서 각 HMW를 평가해줘:
- 제품 관리자 관점: 비즈니스 임팩트가 큰 것은?
- 개발자 관점: 현재 구조에서 구현하기 용이한 것은?
- 디자이너 관점: 사용자 경험을 크게 개선하는 것은?
- 사용자 관점: 지금 당장 가장 필요한 것은?
각 관점에서 총 000 표를 투표를 진행해줘.
종합해서 상위 아이디어를 선별하고 그 이유를 설명해줘."

확인할 것:
- 4가지 관점의 결과가 일치하는가, 다른가?
- 내 직관과 AI의 분석이 같은가, 다른가?
- 다르다면 왜 다른지 이유를 파악했는가?

처음부터 새로 발명할 필요가 없다
- 대부분의 문제는 누군가 이미 해결해본 적이 있다
- 잘 만든 서비스들은 이미 검증된 패턴을 사용한다
- "좋은 예술가는 베끼고, 위대한 예술가는 훔친다" - 피카소

레퍼런스 수집이 중요한 이유
- 검증된 솔루션에서 시작하면 실패 확률이 줄어든다
- 사용자에게 익숙한 패턴은 학습 비용이 낮다
- 다양한 접근 방식을 비교할 수 있다

어떤 레퍼런스를 찾아야 하는가?
- 같은 문제를 해결하는 경쟁 서비스
- 다른 도메인에서 비슷한 패턴을 쓰는 서비스
- UI/UX 패턴 라이브러리 (Mobbin, Dribbble, Behance 등)

"[선정된 HMW 질문]을 해결하기 위한 솔루션을 찾고 싶어.
- 이 문제를 해결하는 다른 유명한 서비스들의 접근 방식을 최대한 많이 조사해줘
- 다른 도메인에서 비슷한 패턴을 사용하는 사례도 최대한 많이 찾아줘
- 위 사례를 검토해 솔루션 아이디어를 최대한 많이 조사해 제안해줘"

확인할 것:
- 제안된 솔루션이 우리 서비스 맥락에 맞는가?
- 사용자에게 익숙한 패턴인가, 새로운 학습이 필요한가?
- 기술적으로 현재 구조에서 구현 가능한가?

"앞에서 수집한 아이디어들을 바탕으로 구체적인 솔루션 3가지를 설계해줘.
각 솔루션은 다음을 포함해줘:
- 솔루션 이름
- 핵심 아이디어
- 주요 화면 구성
- 장단점
- 구현 복잡도 (상/중/하)"

확인할 것:
- 앞에서 모은 아이디어가 솔루션에 반영되었는가?
- 3가지 솔루션이 서로 다른 접근인가?
- 투표로 비교할 수 있을 만큼 구체적인가?

최종 선택의 기준
- 사용자 가치: 문제를 얼마나 효과적으로 해결하는가?
- 구현 가능성: 현재 기술 스택과 시간 내에 만들 수 있는가?
- 학습 가치: 이걸 만들면 무엇을 배울 수 있는가?
- 위험도: 실패했을 때 비용은 얼마인가?

린 스타트업 관점에서 생각하기
- 완벽한 솔루션을 고르는 게 아니다
- "가장 빨리 검증할 수 있는 것"을 고른다
- 작게 만들고, 빨리 배우고, 다시 결정한다

선택했으면 바로 만든다
- 고민하는 시간보다 만들어보는 시간이 더 가치 있다
- AI와 함께면 프로토타입을 빠르게 만들 수 있다

"앞서 설계한 3가지 솔루션 중에서 최종 선택을 해야 해.
다음 기준으로 각 솔루션을 비교 평가해줘:
- 사용자 가치 (1-5점)
- 구현 복잡도 (1-5점, 낮을수록 좋음)
- 검증 용이성 (1-5점)
종합 점수를 계산하고 추천 솔루션을 제안해줘.
선택된 솔루션에 대해 구현 계획을 작성해줘:
이 계획을 /docs/features/ 폴더에 저장해줘."

확인할 것:
- 선택 이유가 논리적인가?
- 구현 계획이 AI에게 명확한 지침이 되는가?
- 성공 지표가 측정 가능한가?

AI 에이전트는 개발자 역할만 하는것이 아니다
- 현황 분석가: 서비스 상황 파악과 우선순위 제안
- UX 리서처: 사용자 여정 지도 작성
- 제품 관리자: 다양한 관점에서 아이디어 평가
- 시장 조사원: 레퍼런스 수집과 경쟁사 분석
- 서비스 기획자: 솔루션 초안 설계와 PRD 작성
- 그리고 개발자: 실제 코드 구현

AI는 전문가 팀이다
- 혼자서 모든 역할을 해야 할 때 AI가 팀원이 되어준다
- 각 역할에 맞는 페르소나를 부여하면 더 좋은 결과가 나온다
- 중요한 건 최종 결정은 항상 내가 한다는 것

"AI가 결정하는 게 아니라, AI의 도움으로 내가 더 좋은 결정을 한다"

- 4가지 시각에서 도출한 HMW 아이디어를 정리한다
- AI와 함께 사용자 여정 지도를 만들어 문제 맥락을 파악한다
- HMW 아이디어들을 그룹화하고 4가지 관점에서 우선순위를 정한다
- 1순위로 선정된 문제에 대해 레퍼런스를 수집한다
- 3가지 솔루션 초안을 설계하고 비교 평가한다
- 최종 솔루션을 선택하고 구현 계획을 작성한다

- 이 한 문장으로 모든것을 설명할 수 있을까?
- 외주 개발자에게 이 한 문장을 전달한다면 내가 원하는 것을 만들 수 있을까?
- 타겟 사용자에게 이 한 문장을 전달한다면 이 서비스가 쓸만하다고 이해할 수 있을까?
- AI에게 이 한 문장을 전달한다면 내가 원하는 소프트웨어가 자동으로 생성될까?

예시: 내가 원하는 요리 사진을 찍으면 자동으로 레시피를 알려주는 서비스를 만들거야
- 외주 개발자: "한식만요? 모든 음식이요? 정확도는 얼마나? 일정은 제가 정해요?"
- 타겟 사용자: "이거 네이버 검색이랑 뭐가 달라요?"
- AI: "웹 앱으로 제작. 회원가입 기능 추가. 레시피는 영문으로 출력."

대상과 가치
- 누구를 위한 서비스인가? 왜 이걸 써야 하는가? 기존 수단과 어떻게 다른가?

범위와 기준
- 어디까지 다루는가? 어떤 품질·정확도가 필요한가? 성공과 실패의 기준은?

구조와 운영
- 어떤 플랫폼·환경에서 작동하는가? 누가 어떻게 접근하는가?
- 무엇을 먼저 만들고 무엇을 뒤로 미루는가?

- AI가 이 서비스의 목적과 방향을 잃지 않도록 하는 가늠좌
- 내가 만들고자 하는 서비스가 해결하는 문제에 대해 스스로 인식하기 쉬운 가이드
- 만들어진 서비스의 전체적인 그림을 예측해 볼 수 있는 청사진

AI는 최고의 전문가지만 '내 마음을 읽는 존재'가 아니다.
AI가 이해하는 것은 오직 내가 제공한 정보와 맥락뿐이다.
따라서 서비스 개요는 'AI가 나의 의도를 정확하게 이해하도록 만드는 첫 번째 데이터셋'이다.

서비스 개요를 작성하는 이유
- 나는 이것을 왜 만드는가?
- 이것은 어떤 문제를 해결하는가?
- 이것을 만들기 위해 알아야 할 것은 무엇인가?

서비스 기획서를 쓸 필요가 없는 이유:
- 기획 문서 = 이미 정해진 내용
- AI가 더 좋은 대안을 제시할 기회를 뺏는다
- AI가 내 생각을 이해하려 하지 않고 문서의 내용만 확인하려 한다
- 기획서가 잘못 만들어 진 것에 대해 피드백을 받는 것이 어렵다

나는 이 서비스를 왜 만들려고 하는지 명확하지 않다
- 만들고 싶은 것은 있지만, 왜 만드는지는 따로 생각해보지 않았다
- 왜 만드는지 안다고 생각했지만, 말로 꺼내보면 생각보다 자세히 설명하기 어렵다
- 만들고 싶은것은 여러개 있는데, 그중 어떤게 진짜인지 모른다

AI가 이 서비스를 왜 만드는 것인지 이해하지 못한다
- 내가 얕게 설명하면 AI도 얕게 이해한다
- 내 의도가 빠진 자리를 AI가 일반적인 답으로 채운다
- 결과물이 그럴듯해 보여서, 어긋난 걸 알아차리기 어렵다

지금까지 쌓은 모든 맥락을 AI에게 전달한다
- 이 서비스가 해결하는 핵심 문제 — 리서치로 검증한 것
- 이 서비스를 사용할 고객의 구체적 상황 — 고객 프로필로 구조화한 것
- 이 솔루션을 선택한 이유와 근거 — 스프린트에서 비교 평가한 것
- 기존 대안과의 차별화 포인트
- 영감을 받은 레퍼런스

- "모든 의견들을 종합해 봤을 때, 이 서비스가 본질적으로 해결하고자 하는 문제는 뭘까?"
- "이런 의견을 가진 사람이 진짜로 해결하고자 하는 문제의 핵심은 뭘까?"
- "이런 서비스가 만들어지면 궁극적으로 어떤 가치를 창출할 수 있을까?"

모든 이야기 끝에 AI와 공감대를 형성하면
- "이 모든 내용들을 바탕으로 서비스의 본질과 핵심에 대해 정리해줘"
- 양식이 거창하거나 맘에들지 않으면 적절하게 양식을 재정의하도록 요청
- 읽으면서 다시한번 스스로 생각해보고, 동의하는지 되묻는다
- 그렇지 않은 부분에 대해 수정하고 다시 정리한다.

사용자는 이미 무언가를 쓰고 있다
- 내 서비스가 해결하려는 문제를 겪는 사람들은 무엇을 하고 있는가?
- 사용자가 느끼는 어려움 - 문제를 해결하고 싶지만 쉽지 않다

차별성이 없으면 존재 이유가 없다
- 기존 방식이 있는데 굳이 새 서비스를 깔아야 하는 이유는 무엇인가?
- "더 좋다"가 아니라, "이것 없이는 안 된다"가 되어야 사용

10배 좋거나, 압도적으로 쉽거나
- 사용자가 기존 방식을 버리고 새 서비스로 옮기는 데는 비용이 든다
- 새로 배워야 하고, 데이터를 옮겨야 하고, 습관을 바꿔야 한다
- 이 비용을 넘어설 만큼의 차이가 있어야 전환한다

차별성의 방향
- 훨씬 빠르게 — 10분 걸리던 것을 1분에
- 훨씬 쉽게 — 5단계를 1단계로
- 훨씬 싸게 — 월 5만원을 무료로
- 아예 불가능했던 것을 가능하게

"우리 서비스의 차별점은 뭐야?"라고 물으면 그럴듯한 마케팅 문구가 나온다

직접 경쟁 서비스
- 같은 문제를 같은 방식으로 풀고 있는 서비스
- 앱스토어 검색, 웹 검색으로 찾을 수 있다

간접 대안
- 엑셀, 메모장, 카카오톡 그룹채팅, 수기 관리
- 사용자가 "서비스를 안 쓰고도 해결하고 있는 방법" 전부가 경쟁 상대다

AI와 웹 검색을 함께 활용한다
- AI는 이 분야의 서비스 생태계를 폭넓게 알고 있다 — 먼저 물어본다
- 웹 검색으로 실제 사용자 리뷰와 평가를 확인한다 — 마케팅 문구가 아니라 진짜 반응을 본다
- 최고의 방법 — 내가 직접 사용해보기

대안 탐색
- "내 서비스가 해결하려는 문제를 사용자들이 현재 어떤 방법으로 해결하고 있는지 조사해줘. 전문 서비스뿐 아니라 엑셀, 메신저, 수기 관리 등 비공식적 방법도 포함해서 찾아줘"
- "각 대안의 장점과 한계를 정리하고, 실제 사용자들이 불만을 느끼는 지점을 웹에서 찾아서 정리해줘"

차별성 검증
- "내 서비스 개요에 적힌 해결 방안이 기존 대안 대비 어떤 점에서 명확하게 나은지 비교해줘. 차별성이 약하다면 어떤 방향으로 강화할 수 있는지도 제안해줘"

"앞서 조사한 내용들을 바탕으로 이 MVP를 만드는 과정에서 목적과 의도를 유지하는데 필요한 모든 내용이 포함되어있는 서비스 개요를 작성해줘. 이 서비스에 대해 아무것도 모르는 사람이 읽어도 필요성과 방향을 이해할 수 있어야 해"
